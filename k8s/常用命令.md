
### 创建资源
```
kubectl create -f config.yaml
```

| **特点** | **说明**                           |
| ------ | -------------------------------- |
| **用途** | **仅用于创建**新的资源对象。                 |
| **行为** | 如果资源已存在，命令会**失败并报错**，不会尝试更新现有资源。 |

```
kubectl apply -f config.yaml
```

| **特点**   | **说明**                                                                                                                                   |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| **用途**   | 用于**创建或更新**资源对象。                                                                                                                         |
| **行为**   | 1. **如果资源不存在：** 它会创建新资源。 2. **如果资源已存在：** 它会将其定义与配置文件 (`config.yaml`) 进行比较，并仅应用配置文件中指定的**更改**，从而更新现有资源。它还会保留资源上**其他工具**（如 HPA 自动扩缩）所做的更改。 |
| **管理方式** | **声明式管理** (Declarative Management)。您告诉 Kubernetes **您期望的最终状态**（“资源应长成这样”），Kubernetes 会负责实现它。                                             |
| **适用场景** | **推荐**在生产环境和所有需要持续管理和更新资源时使用。允许您安全地重复执行（幂等性）。                                                                                            |
|          |                                                                                                                                          |
### 日志
最基础的方式：  
`kubectl logs <pod-name>`

如果 Pod 里有多个容器，加个 `-c`：  
`kubectl logs <pod-name> -c <container-name>`

如果你想一路看着日志滚动，就像盯着一条不断写字的纸卷：  
`kubectl logs -f <pod-name>`
`kubectl logs -f <pod-name> -c <container-name>`

如果 Pod 重启过，你想看看它“上一个生命”的遗言：  
`kubectl logs <pod-name> --previous`

在某些场景下 Pod 太多，先用 label 过滤再看日志比较文明：  
`kubectl logs -l app=myapp --all-containers=true`
